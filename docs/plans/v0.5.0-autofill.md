# v0.5.0 — Auto-fill from user profile

Status : **draft**, not started.

## Goal

Let the user fill a PDF in **one click** using a saved profile (name, email, …), the selected signature and the selected paraph. Reduces the most tedious step of "fill & sign" to a single button.

The killer use case : the user fills the same kind of admin form (rental application, expense claim, school enrolment) every few weeks. Manual filling is repetitive — auto-fill turns it into "open + click + review + sign + export".

## Scope

### In (v0.5.0)
- Persistent **user profile** (key/value pairs : `firstName`, `lastName`, `email`, `phone`, `address`, `city`, `zip`, `country`, `iban`, `dateOfBirth`, …). Stored locally via the existing `usePersistentState` + Tauri command pair (no localStorage fallback for now — same constraint as signatures/paraphs).
- **Profile management UI** in the sidebar (collapsible section). Each field is a row with a label and a value input. User can add custom keys.
- **"Auto-fill" toolbar button**. Computes an `AutoFillPlan` and applies it.
- **Detection on AcroForm PDFs** : layers 1 + 2 from the discussion :
  - L1 : field name + `alternativeText` matched against a multilingual dictionary
  - L2 : nearby label extracted from `getTextContent()`, spatially correlated with each field's rect
- **Signature field detection** : pdf.js exposes `fieldType === "Sig"`. Place a `SignatureItem` overlay at the widget's rect, using the currently-selected signature.
- **Paraph zone heuristic** : detect via field-name regex + spatial repetition across pages. When found, create the `Paraph` master at that rect (one paraph master ⇒ rendered on every page via existing infra).
- **Preview before applying** : modal showing "12 text fields / 1 signature / 1 paraph detected — apply ?" with the count of unmatched fields. User can confirm or cancel.

### Out (deferred to v0.6.0+)
- Flat-PDF / PDF/A support (no AcroForm). Same detection ideas but using `getTextContent()` patterns + spatial heuristics ; produces overlay items instead of form values.
- OCR for scanned PDFs (Tesseract.js or native Tesseract).
- LLM-based detection for ambiguous cases. Should be **opt-in**, **off by default**, and ideally local (Ollama).
- Profile import/export (vCard, JSON).
- Multiple profiles (e.g. "Personal" / "Work").
- Smart formatting (date format per locale, phone number country code, …).

## Architecture

### Data model

```ts
// src/types.ts
export type ProfileEntry = {
  key: string;        // canonical key, e.g. "email"
  label: string;      // human label, e.g. "Email"
  value: string;
};

export type Profile = ProfileEntry[];
```

Keys are user-controlled strings but a **canonical set** is suggested at first launch (the dictionary below).

### Detection output

```ts
// src/autofill/types.ts
export type AutoFillPlan = {
  /** Native AcroForm values keyed by field name. */
  formValues: FormValues;
  /** Signature items to push into `items[]` (one per detected signature field). */
  newItems: Item[];
  /** Paraph master to set (null = none detected). */
  paraph: Paraph | null;
  /** Fields the engine could not confidently match — surfaced for UX. */
  unmatched: FieldDescriptor[];
  /** Statistics for the preview modal. */
  stats: {
    matchedText: number;
    matchedChoice: number;
    matchedCheckbox: number;
    matchedSignature: number;
    matchedParaph: number;
    skipped: number;
  };
};
```

### New module : `src/autofill/`

```
src/autofill/
  dictionary.ts          // multilingual key aliases (see below)
  matchFieldName.ts      // L1 : exact / fuzzy match on field name + alternativeText
  matchByLabel.ts        // L2 : getTextContent + spatial proximity
  detectSignature.ts     // FieldType === "Sig" handling
  detectParaph.ts        // spatial repetition + name regex heuristic
  buildPlan.ts           // composes the layers into an AutoFillPlan
  types.ts
  *.test.ts
```

`buildPlan(doc, fields, profile, selectedSignature, selectedParaph)` is the public entry point ; everything else is internal.

### Dictionary (sketch)

```ts
// Each canonical key maps to a list of name/label aliases across locales.
export const FIELD_DICTIONARY: Record<string, string[]> = {
  firstName: ["firstname", "first_name", "given_name", "prenom", "prénom", "vorname", "nombre", "имя", "名"],
  lastName:  ["lastname", "last_name", "surname", "family_name", "nom", "nachname", "apellido", "прізвище", "姓"],
  email:     ["email", "e-mail", "mail", "courriel", "correo"],
  phone:     ["phone", "tel", "telephone", "téléphone", "mobile", "telefon"],
  address:   ["address", "adresse", "calle", "anschrift"],
  city:      ["city", "ville", "stadt", "ciudad"],
  zip:       ["zip", "postcode", "code_postal", "cp", "postal_code", "plz"],
  // …
};
```

Matching is **case-insensitive** with a normalizing pass (strip diacritics, collapse spaces/underscores/dashes).

### Paraph heuristic

Detect via either signal :
1. **Name regex** on field name or alternativeText : `/paraph|paraf|initial|init|vise/i`
2. **Spatial repetition** : group text-field widgets by approximate rect (`±5pt` on x, y, w, h). If the same rect (signature) appears on `≥ 2` pages (and the doc has more than 2 pages), it's almost certainly a paraph.

When both signals agree, confidence is high. When only one signal fires, surface in the preview as "Detected paraph candidate ? (yes / no)".

### UI changes

- **Sidebar : new "Profile" section** above Signatures. Same visual treatment as the existing sections. Each entry is `label + value input + delete`. A `+` button adds a new row. A "Reset to defaults" button repopulates the canonical set.
- **Toolbar : new "Auto-fill" button** (icon : `MagicWand` from Phosphor) between the existing tool group and the ink picker. Disabled when no PDF is open. On click, opens the preview modal.
- **Preview modal** : count of detected matches, list of detected mappings (`Field name → Profile value`), unmatched fields listed at the bottom, "Apply" / "Cancel" buttons.

## Risks / open questions

- **Field-type mismatch** : the profile has a string for `dateOfBirth` ("1990-01-15") but the PDF field may be a text field expecting `DD/MM/YYYY`. v0.5.0 just writes the raw string ; smart formatting is v0.6.0+.
- **Checkbox auto-fill** : a profile doesn't really carry checkbox values. Skip checkboxes for now (they stay manual). Same for radio groups unless one option matches a profile value exactly.
- **Paraph false positives** : if a footer has two paraph candidates (e.g. signature + paraph on the same page), the spatial-repetition rule could fire on either. The preview modal lets the user catch this.
- **Performance** : `getTextContent()` is fast per page, but doing it for every page on every "Auto-fill" click could be noticeable on long docs. Cache the result per opened document.
- **Existing test breakage** : surfacing signature widgets changes `useFormFields` behaviour — be deliberate about the test update.

## Decisions

- **Default profile** : ship the canonical set pre-populated with empty values (firstName, lastName, email, phone, address, city, zip, country, dateOfBirth, iban). User sees what to fill at a glance ; the input rows act as their own discovery hint.
- **Auto-fill behaviour** : **overwrite matched fields only**. Fields not matched by the engine are left untouched — any prior manual edits survive.
- **Button position** : near Export PDF / Print in the toolbar's document-wide-actions cluster. Conceptually a one-shot doc-wide write, not a tool mode.
