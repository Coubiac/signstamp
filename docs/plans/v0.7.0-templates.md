# v0.7.0 — User-trained templates

Status : **draft**, not started.

## Goal

Let the user save the current set of overlays (text, signature, paraph,
check, date, …) as a **named template**, then re-apply that template
to **any future PDF** with one click. Built to be reliable on 100% of
PDFs — no detection magic.

The canonical use case : an admin or freelancer who fills the same
flavour of contract every week. The first time, they place the
overlays by hand (3-5 min). Every subsequent fill is one click.

## Scope (v0.7.0)

### In
- Save the current document state (`items` + `paraph`) as a template
  with a user-chosen name.
- List, rename, delete templates from a dedicated UI.
- Apply a chosen template to the currently-open PDF — overlays appear
  at their stored positions with their stored values.
- **Date auto-refresh** : items added via the "date" tool carry an
  `autoDate` flag ; when a template is applied, those items' values
  are recomputed to today's date (in the active locale).
- Persistence : Tauri-only storage (`templates.json` in `app_data_dir`).

### Out (deferred to v0.8.0+)
- Automatic detection / suggestion of which template fits the open PDF.
- Slots / placeholders ("name to fill in at apply time").
- Importing / exporting templates (sharing between users).
- Per-template asset autonomy (templates currently reference
  signature / paraph asset IDs — if the asset is deleted, the
  overlay becomes orphaned and the UI surfaces a warning).

## Decisions (locked in)

- **PDF identification** : none. The user picks which template to
  apply from a list, regardless of which PDF is open. Sidesteps the
  "is this the same PDF ?" problem entirely.
- **Trigger** : a manual "Apply template" button in the toolbar
  (plus a File menu entry). No auto-apply.
- **Multiple templates per use case** : yes. A user can save "contrat
  NEXT2 — perso" and "contrat NEXT2 — pro" as two separate templates.
- **Content** : fixed values per item ; the user duplicates a template
  if they want a variant. Exception : date items auto-refresh.

## Data model

```ts
// src/types.ts (extension)
export type TextItem = {
  // …existing fields…
  /** True for items added via the date tool. When a template is
   *  applied, the value of these items is replaced with today's
   *  date in the active locale. */
  autoDate?: boolean;
};

// New module : src/templates/types.ts
export type Template = {
  id: string;
  name: string;
  /** ISO timestamp ; used to sort and to show "last edited" in the UI. */
  updatedAt: string;
  items: Item[];
  paraph: Paraph | null;
};
```

### New module : `src/templates/`

### Tauri commands (mirror the existing signatures pair)

`StoredTemplate` mirrors the frontend shape ; the JSON file lives at
`<app_data_dir>/templates.json`.

### Frontend hook

```ts

```

Reuses `usePersistentState` + `tauriOnlyAdapter` (templates can be
substantial — Tauri-only avoids polluting localStorage).

### UI surfaces

- **Toolbar button** : a "Templates" icon between Print and Export.
  Opens the templates modal.
- **File menu** : two entries — `Save as template…` (`Cmd/Ctrl+T`)
  and `Templates…` (open the modal).
- **Templates modal** : a list of saved templates with apply / rename
  / delete actions, plus a "Save current as template" header button.

### Apply logic

```ts
// src/templates/applyTemplate.ts
export type ApplyTemplateArgs = {
  template: Template;
  locale: Locale;          // for autoDate formatting
  now?: Date;              // injection point for tests
};

export type ApplyResult = {
  items: Item[];           // ready to push into items state
  paraph: Paraph | null;   // ready to setParaph()
};

export function applyTemplate(args: ApplyTemplateArgs): ApplyResult {
  const now = args.now ?? new Date();
  const items = args.template.items.map((it) => {
    const cloned: Item = { ...it, id: uid() };
    if (cloned.type === "text" && cloned.autoDate) {
      cloned.value = formatLocaleDate(args.locale, now);
    }
    return cloned;
  });
  return { items, paraph: args.template.paraph };
}
```

Notes :
- IDs are regenerated so applying the same template twice doesn't
  create id collisions.
- The host (`App.tsx`) decides how to merge : **append** (default,
  add to existing items) or **replace** (clear items first then
  apply). v1 defaults to append ; we may add a toggle later.

### Date tool wiring

Currently `App.tsx`'s "date" tool adds a regular `TextItem` whose
`value` is `formatLocaleDate(lang, new Date())`. The fix : also
set `autoDate: true` on that item at creation time.

```ts
// In the date-tool placement branch
appendItem({
  // …existing fields…
  value: formatLocaleDate(lang, new Date()),
  autoDate: true
});
```

Existing date items (saved in undo history or older sessions) won't
have the flag and behave as static text — acceptable.

## Phasing (order of commits)

1. `Template` type + `autoDate` flag on `TextItem` + Tauri commands +
   `useTemplates` hook.
2. `applyTemplate` pure function + unit tests (clone, id regen,
   date refresh, paraph carry).
3. Date tool sets `autoDate: true` on the item it creates.
4. Templates modal (list + apply + delete + rename) + "Save current"
   modal.
5. Toolbar button + File menu entries + keyboard shortcuts.
6. End-to-end smoke : open a PDF, place overlays, save as template,
   open another PDF, apply, verify.

- Existing 113 tests must stay green.

## Risks / open questions

- **Orphaned asset references** : a saved template references signature
  / paraph IDs by `id`. If the user deletes the asset later, the
  overlay can't render. Mitigation : at apply time, check that all
  referenced assets still exist ; surface unresolved ones in a
  warning and skip those items.
- **Asset bytes vs reference** : we chose references (lighter, simpler)
  over baking asset bytes into the template. Trade-off : a template
  isn't fully portable. Revisit if export/import becomes a feature.
- **Append vs replace at apply** : v1 appends. If the user applies
  twice by mistake, duplicates appear. Acceptable as long as Undo
  still works (it does — append is a single recorded action).
- **Locale at apply time** : `autoDate` uses the locale active **at
  apply time**, not at save time. Intended : a user who switches the
  app to English keeps applying with English-formatted dates.
